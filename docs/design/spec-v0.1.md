## 1. Decyzja

`django-dev-helpers` to **Django app** wystawiająca trzy ortogonalne funkcje dev-time:

1. **Autologin endpoint** — view, która po podaniu poprawnego tokenu loguje admina i otwiera stronę. Eliminuje ręczne wpisywanie `admin/admin` przy każdym restarcie i pozwala agentom LLM (Claude Code, Aider, Cursor) wykonywać `WebFetch` na zalogowane strony.

2. **Dotfile'y w `project_root`** — krótkie pliki tekstowe (`.dev_helpers_token`, `.dev_helpers_port`, `.dev_helpers_pg_port`, `.dev_helpers_redis_port`) z aktualnym tokenem i portami. Agent czyta `cat .dev_helpers_token` zamiast parsować banner.

3. **Agent help / prompt template** — gotowy do skopiowania snippet z curl/psql/redis-cli wzorcami, drukowany przez `manage.py dev_helpers_print_help` albo automatycznie przy starcie runservera.

Plus:

4. **Gitignore self-check** — sanity check, że dotfile'y są w `.gitignore`. Tylko warning + opcjonalny auto-add, nigdy commit.

Pakiet **nie ma** zależności od `django-run-site` ani testcontainers. Współpraca odbywa się przez kontrakt env-var (§13).

---

## 2. Cel

Jedna paczka, którą można dodać do dowolnego projektu Django w trybie dev, żeby:

- agent kodujący umiał pobierać zalogowane strony bez interaktywnego logowania,
- developer nie musiał szukać aktualnych portów po `docker ps`,
- dev-stack drukował copy-paste'owalny prompt do paste'owania w sesji LLM,
- `.gitignore` nie zostawiał tokenu w commicie przez przeoczenie.

**Standalone use case:** projekt z `docker-compose.yml` definiującym PostgreSQL/Redis na ustalonych portach, gołe `manage.py runserver` na hoście, `.env` z DB connection params. `pip install django-dev-helpers` + dodanie do `INSTALLED_APPS` daje wszystkie powyższe funkcje bez zmiany dev workflow.

**W połączeniu z `django-run-site`:** `django-run-site` orkiestruje testcontainery i ustawia env-y, `django-dev-helpers` AppConfig je odczytuje i robi swoje. Brak bezpośredniej zależności w kodzie.

---

## 3. Non-goals v0.1

- Nie dostarczamy CLI (to robi `django-run-site` lub user własnym skryptem).
- Nie startujemy testcontainerów ani innych external services.
- Nie zarządzamy migracjami, superuser-em ani Celery.
- Nie modyfikujemy URL-conf przez monkey-patch — user explicit dodaje `autologin_urlpatterns()` w swoim `urls.py`.
- Nie wspieramy multi-tenancy ani per-request token rotacji.
- Nie ma session-based autologin — używamy token-w-querystringu, który po dopasowaniu wykonuje `login(request, user)`.

## 4. Bezpieczeństwo — kill switch dla produkcji

**To jest najważniejsza sekcja tej spec'i.** Autologin endpoint to backdoor dla admina. Pakiet musi być **default-off na produkcji**.

### 4.1 Reguła aktywacji — explicit precedence

Aktywacja jest zawsze explicit. `DEBUG=True` **nie wystarczy**.

**Precedence (od najwyższego do najniższego):**

1. `settings.DJANGO_DEV_HELPERS["enabled"] = False` w `settings.py` → **hard override, pakiet wyłączony**, niezależnie od env-u. Dev w team-ie który nie chce dev-helpers ustawia to raz w settings.local i zapomina.
2. `settings.DJANGO_DEV_HELPERS["enabled"] = True` w `settings.py` → włączony.
3. `DJANGO_DEV_HELPERS_ENABLED=1` env var → włączony.
4. brak żadnego z powyższych → wyłączony (default).

W praktyce: settings False = always off, settings True = always on, env = on jeśli settings nie zdecydowała.

```python
# conf.py
def is_active(settings_dict, env):
    cfg_value = settings_dict.get("enabled", None)
    if cfg_value is False:
        return False  # hard override
    if cfg_value is True:
        return True
    return env.get("DJANGO_DEV_HELPERS_ENABLED") == "1"
```

`cfg_value is None` (default — user nie ustawił `enabled` w settings) → spadamy do env-u.

### 4.2 Zachowanie gdy nieaktywne

- Autologin URL pattern **nie jest rejestrowany** przez `autologin_urlpatterns()` — dla URL resolver-a nie istnieje (zwykłe Django 404 z URL resolution path-a, nie nasze defensywne `Http404`).
- AppConfig.ready() exit early — nie pisze dotfile'i, nie spawnuje thread-ów, nie drukuje warningów.
- `manage.py dev_helpers_print_help` zwraca błąd "django-dev-helpers nie jest aktywne; ustaw DJANGO_DEV_HELPERS_ENABLED=1".
- Gitignore self-check się nie odpala.

Jednolite zachowanie: pakiet jest jakby go nie było. Brak hidden 404 view, brak partial side effects.

### 4.3 Hard fail na produkcji

Gdy pakiet wykryje aktywację (`enabled=True`) **i** `settings.DEBUG=False` **i** uruchamiane jest `runserver` (lub WSGI/ASGI handler) — **`AppConfig.ready()` rzuca `ImproperlyConfigured`**. Django nie startuje. Brak override-u.

**Ważna kolejność:** raise następuje **tylko po** `is_active()` check. Jeżeli pakiet nie jest aktywny (`enabled=False`, `DJANGO_DEV_HELPERS_ENABLED` nieustawiony) — `AppConfig.ready()` zwraca early, bez warnings, bez raises. Spec §9.1 pokazuje pełny flow.

**Ważne ograniczenie:** raise dzieje się **tylko** gdy serwer faktycznie wystawia HTTP endpoint (`runserver`, `runserver_plus`, WSGI/ASGI). Inne management commands (`migrate`, `collectstatic`, `dbshell`, `dev_helpers_doctor`, `test`) — backdoor nieaktywny, więc raise nie ma sensu i mógłby blokować legitimate dev/CI workflow.

```python
# safety.py
import sys
import os
from django.core.exceptions import ImproperlyConfigured

# Whitelist znanych management commands, które NIE wystawiają HTTP endpointu.
# Reszta — łącznie z gunicorn/uwsgi/daphne/uvicorn/asgi/wsgi handler-ami,
# nieznanymi runner-ami i custom WSGI app-ami — traktowana jak "serving"
# (fail-closed): jeśli nie wiemy, zakładamy że backdoor może być aktywny.
_NON_SERVING_COMMANDS = {
    "migrate", "makemigrations", "showmigrations", "sqlmigrate",
    "collectstatic", "compilemessages", "makemessages",
    "test", "check", "diffsettings",
    "shell", "dbshell",
    "createsuperuser", "changepassword",
    "dumpdata", "loaddata", "flush", "inspectdb",
    "dev_helpers_doctor", "dev_helpers_print_help",
    "dev_helpers_check_gitignore",
    "help", "version",
}

def _is_serving():
    """True gdy proces wystawia (lub może wystawiać) HTTP endpoint.

    Fail-closed: wszystko, czego nie rozpoznamy jako non-serving
    management command — traktujemy jako serving. Gunicorn / uwsgi /
    daphne / uvicorn / Apache mod_wsgi / custom WSGI runner pasują do
    tej kategorii, bo `sys.argv[1]` nie jest naszym whitelist-owanym
    poleceniem.
    """
    # Override dla testów defensywnych — wymuszenie checku w CI.
    if os.environ.get("DJANGO_DEV_HELPERS_FORCE_SAFETY_CHECK") == "1":
        return True
    # Empty argv (mało prawdopodobne) → serving (fail-closed).
    if len(sys.argv) < 2:
        return True
    # Pierwszy argument po `manage.py` (lub po nazwie wrapper-a).
    cmd = sys.argv[1]
    return cmd not in _NON_SERVING_COMMANDS

def assert_safe_to_activate(cfg):
    """Wywoływane TYLKO po `cfg.is_active() == True`."""
    from django.conf import settings
    if not _is_serving():
        # migrate / collectstatic / doctor / test — backdoor nieaktywny.
        # Nie raisuj, nie blokuj. Doctor sam zdiagnozuje dev-vs-prod.
        return
    if not settings.DEBUG:
        raise ImproperlyConfigured(
            "django-dev-helpers cannot be enabled when settings.DEBUG=False "
            "and serving HTTP. This package exposes an autologin backdoor "
            "and is not safe for production. To use in dev: set DEBUG=True. "
            "To disable: remove DJANGO_DEV_HELPERS_ENABLED env var or set "
            "DJANGO_DEV_HELPERS['enabled'] = False."
        )
```

**Fail-closed** to świadoma decyzja: gunicorn, uwsgi, mod_wsgi, daphne, uvicorn, custom WSGI runner — wszystkie te procesy wystawiają HTTP, ale ich `sys.argv` nie pasuje do whitelist-y serving commands. Whitelistując znane *non-serving* commands, default behavior dla nieznanych = "treat as serving" = run the safety check. Lepiej raisować false-positive na nieznanym command niż przepuścić backdoor na produkcję.

Lista `_NON_SERVING_COMMANDS` pokrywa standardowe Django management commands. Custom commands (dev-helpers własne, projektowe) — automatycznie traktowane jako serving (fail-closed) chyba że dopisane explicit. To jest celowe — projekt dodający custom command może go dodać przez `DJANGO_DEV_HELPERS["safety"]["non_serving_commands"]` (extra whitelist).

Override dla CI / testów defensywnych: `DJANGO_DEV_HELPERS_FORCE_SAFETY_CHECK=1` wymusza raise check nawet w non-serving commands. Użyteczne, gdy user chce upewnić się że workflow z `DEBUG=False` nie ma `enabled=True` przez pomyłkę (CI pre-flight). Nazwa zmiennej (`FORCE_SAFETY_CHECK`, nie `FORCE_CHECK`) podkreśla że to wymusza *wykonanie* sprawdzenia, nie *przepuszczenie* backdoor-a — to nie jest override produkcyjny.

Inne sanity checks (warning, nie exception):

- `ALLOWED_HOSTS` zawiera coś innego niż `localhost`, `127.0.0.1`, `*` → `RuntimeWarning` (info, nie exception — staging może mieć custom hostname przez `/etc/hosts` redirect, ale dalej być devem; warto głośno powiedzieć),
- `SECRET_KEY` krótszy niż 50 znaków lub bez prefiksu `django-insecure-` → info (sanity w drugą stronę: SECRET_KEY wygląda jak production secret).

### 4.4 Refusal w widoku

Mimo `enabled=True`, autologin view dodatkowo:

- weryfikuje obecność tokenu w env (`DEV_HELPERS_AUTOLOGIN_TOKEN`); brak → `Http404`,
- używa `hmac.compare_digest` do porównania tokenu (timing-safe),
- weryfikuje hostname requestu: hostname z `request.get_host()` (bez portu, IPv6-aware) musi być w whitelist.

**Hostname extraction**: użyj `django.http.request.split_domain_port(request.get_host())[0]` żeby odciąć port. `request.get_host()` zwraca `localhost:8000` — bezpośrednie porównanie z `"localhost"` nie pasowało. IPv6 hosty są w nawiasach (`[::1]:8000` → hostname `[::1]`); `split_domain_port` to obsługuje.

Default whitelist (zawsze): `localhost`, `127.0.0.1`, `[::1]`. Dodatkowy whitelist z configu: `DJANGO_DEV_HELPERS["autologin"]["allowed_hosts"]` (lista). Wzorce glob (`*.localhost`, `*.ngrok.io`) dopuszczone — match przez `fnmatch.fnmatchcase` po lowercase-owaniu obu stron.

```python
# views.py fragment
from django.http.request import split_domain_port

hostname, _ = split_domain_port(request.get_host())
hostname = hostname.lower()
default_allow = {"localhost", "127.0.0.1", "[::1]"}
extra_allow = cfg.autologin.allowed_hosts
if hostname not in default_allow:
    if not any(fnmatch.fnmatchcase(hostname, pat.lower()) for pat in extra_allow):
        raise Http404()
```

Whitelist hostów jest świadomie restrykcyjna. Use cases wymagające szerszego dostępu:

- LAN IP (np. testowanie z telefonu pod `192.168.1.42:8000`) → user dodaje `"192.168.1.42"` do `allowed_hosts`,
- ngrok / Cloudflare Tunnel (`abc.ngrok.io`) → user dodaje `"*.ngrok.io"` (wzorzec glob),
- Docker container LAN name → user dodaje konkretny hostname.

HTTPS-specific checks (np. `SECURE_SSL_REDIRECT`) — out of scope w v0.1, bo dev-stack `django-run-site` nie wystawia HTTPS na runserverze. Gdy v0.x doda HTTPS support, defense in depth dla HTTPS dokleimy wtedy.

Defense in depth: nawet gdyby user przez pomyłkę odpalił to na produkcji (przy `DEBUG=True`, omijając §4.3), każdy z tych warunków odetnie endpoint w 99% scenariuszy.

## 7. Konfiguracja — `settings.DJANGO_DEV_HELPERS`

```python
# settings.py
DJANGO_DEV_HELPERS = {
    "enabled": True,  # alternatywa: env DJANGO_DEV_HELPERS_ENABLED=1

    "autologin": {
        "enabled": True,
        # Identyfikator usera do zalogowania.
        "user_lookup_field": "username",   # albo "email", "pk", custom field
        "user_lookup_value": "admin",      # wartość do dopasowania
        "url_path": "__autologin__/",
        "redirect_to": "/",
        "auth_backend": "django.contrib.auth.backends.ModelBackend",
        "flash_message": "",
        # Domyślnie pusta lista — nie ustawiamy żadnych cookies poza Django
        # session cookie ustawianym przez login(). Projekty używające
        # cookie-banner libraries (django-cookielaw, django-cookie-consent)
        # mogą dodać tutaj cookie zwalniające z banneru w dev. Patrz §15
        # (BPP migration) dla przykładu.
        "extra_cookies": [],
        # Dodatkowy whitelist hostów; localhost i 127.0.0.1 są zawsze OK.
        "allowed_hosts": [],
    },

    "dotfiles": {
        "enabled": True,
        # None = auto-detect (patrz §9.5 — Project root discovery).
        "directory": None,
        "token_filename": ".dev_helpers_token",
        "port_filename": ".dev_helpers_port",
        "pg_port_filename": ".dev_helpers_pg_port",
        "pg_host_filename": ".dev_helpers_pg_host",
        "redis_port_filename": ".dev_helpers_redis_port",
        "redis_host_filename": ".dev_helpers_redis_host",
        "token_chmod": 0o600,
    },

    "agent_help": {
        # Auto-print prompt do stdout przy starcie runservera (na first request).
        "auto_print": True,
        # Custom template (None = default)
        "template": None,
        # Hostname wstawiany do prompt-a. None = auto: gdy bind 0.0.0.0/::,
        # substytucja na localhost; inaczej użyj wartości bind. Override np.
        # "myhost.local" gdy testujemy w LAN.
        "display_host": None,
        # Czy pokazywać DB password w prompt template. Default True (dev
        # password jest zazwyczaj "password" — nie jest sekretem). User
        # wyłącza dla custom dev DB-i z realnymi credentials.
        "show_db_credentials": True,
    },

    "browser_open": {
        # Otwiera browser z URL autologin (z tokenem) po self-probe.
        # Default True — to jest kluczowy dev convenience.
        # W trybie z django-run-site: CLI też otwiera browser (homepage),
        # więc user dostaje dwie zakładki: homepage + autologin URL.
        # User wyłącza CLI browser przez --no-browser; dev-helpers wyłącza
        # przez "enabled": False tutaj.
        "enabled": True,
        # None = auto: gdy autologin.enabled → autologin URL z tokenem;
        # inaczej "/" (homepage). User może override (np. "/admin/").
        "url_path": None,
        "probe_path": "/admin/login/",
        "probe_timeout_seconds": 30.0,
    },

    "gitignore": {
        "mode": "warn",  # warn | auto-add | error | off
        "path": None,    # None = project_root/.gitignore
    },

    # Pull DB/Redis info dla dotfile'i + prompt template:
    "lookup": {
        # "settings" = czytaj z settings.DATABASES + settings.CACHES
        # "env"      = czytaj z env vars DEV_HELPERS_DB_*, DEV_HELPERS_REDIS_*
        # "auto"     = env wins jeśli ustawione, inaczej settings
        "source": "auto",
        # Custom override: lookup callable. Sygnatura:
        #   def resolve(cfg) -> dict[str, Any]
        # Zwracany dict może zawierać klucze: pg_host, pg_port, db_user,
        # db_password, db_name, redis_host, redis_port. Brakujące klucze
        # → fallback do default lookup (env/settings).
        "callable": None,  # "myproject.dev_helpers_lookup:resolve"
    },
}
```

`conf.py` merguje z defaultami (każde brakujące pole bierze default), waliduje typy i enumy, raisuje `ImproperlyConfigured` przy konflikcie.

### 7.1 Minimalny config

Najmniejsza konfiguracja w `settings.py`:

```python
INSTALLED_APPS = [..., "django_dev_helpers"]

DJANGO_DEV_HELPERS = {"enabled": True}
```

Default user lookup: `username="admin"`. Dla projektów z custom user modelem (np. `email` jako USERNAME_FIELD) trzeba ustawić:

```python
DJANGO_DEV_HELPERS = {
    "enabled": True,
    "autologin": {
        "user_lookup_field": "email",
        "user_lookup_value": "admin@example.com",
    },
}
```

Plus w `urls.py`:

```python
from django_dev_helpers.urls import autologin_urlpatterns

urlpatterns = [
    *autologin_urlpatterns(),
    path("admin/", admin.site.urls),
    # ...
]
```

To wystarczy. Wszystko inne ma sensowne defaulty.

## 8. Autologin endpoint

### 8.1 URL pattern

```python
# django_dev_helpers/urls.py
def autologin_urlpatterns():
    """Zwraca listę URL patterns. Używane: *autologin_urlpatterns() w urls.py."""
    from .conf import get_config
    cfg = get_config()
    if not cfg.is_active() or not cfg.autologin.enabled:
        return []
    return [
        path(cfg.autologin.url_path, views.autologin, name="dev_helpers_autologin"),
    ]
```

Gdy pakiet nieaktywny — funkcja zwraca pustą listę. URL nie istnieje. **Nie zwraca patterna który robi 404**, tylko nie rejestruje patternu w ogóle. Cleaner.

### 8.2 View flow

```python
def autologin(request):
    cfg = get_config()
    cfg.refuse_if_inactive()  # raise Http404 jeśli !is_active()
    cfg.refuse_if_unsafe_host(request)  # patrz §4.4

    expected = os.environ.get("DEV_HELPERS_AUTOLOGIN_TOKEN")
    if not expected:
        raise Http404()

    provided = request.GET.get("token") or ""
    if not hmac.compare_digest(expected, provided):
        raise Http404()

    User = get_user_model()
    try:
        user = User.objects.get(
            **{cfg.autologin.user_lookup_field: cfg.autologin.user_lookup_value}
        )
    except User.DoesNotExist as exc:
        raise Http404() from exc
    except User.MultipleObjectsReturned as exc:
        # Lookup field nie jest unique — niepoprawna konfiguracja.
        raise Http404() from exc

    login(request, user, backend=cfg.autologin.auth_backend)

    if cfg.autologin.flash_message:
        from django.contrib import messages
        messages.success(request, cfg.autologin.flash_message)

    response = HttpResponseRedirect(cfg.autologin.redirect_to)

    for cookie in cfg.autologin.extra_cookies:
        response.set_cookie(**cookie)

    return response
```

Wszystkie negatywne ścieżki to `Http404` — endpoint zachowuje się jak nieistniejący. Brak `403 Forbidden` ani `401 Unauthorized`, żeby nie sygnalizować, że URL jest "ważny ale niedostępny".

### 8.3 Token: env var vs self-generated

**W trybie z `django-run-site`:**

CLI generuje token (`secrets.token_urlsafe(32)`) i ustawia env var `DEV_HELPERS_AUTOLOGIN_TOKEN` dla procesu runservera. Django app czyta env, view porównuje. AppConfig zapisuje token do `.dev_helpers_token`.

**Standalone:**

Env var nie istnieje → AppConfig sam generuje (jeśli `autologin.enabled=true`) i zapisuje do dotfile'a. View używa tego samego mechanizmu (env var ustawiana przez AppConfig na bieżący proces).

### 8.4 Autoreloader handling

Django runserver z autoreload spawnuje child Python process. Oba dziedziczą env vars od parenta. Ale: jeśli tokenu nie było w parent env i parent AppConfig wygenerował token, to child AppConfig też wygeneruje swój token (różny). View w child-u nie zaakceptuje tokenu z dotfile'a parenta.

Rozwiązanie: AppConfig sprawdza, czy `DEV_HELPERS_AUTOLOGIN_TOKEN` jest już w env. Jeśli tak — nie generuje, używa istniejącego. Jeśli nie — generuje, **i ustawia w `os.environ`**, żeby fork/child dziedziczył.

```python
# tokens.py
def init_token() -> str:
    existing = os.environ.get("DEV_HELPERS_AUTOLOGIN_TOKEN")
    if existing:
        return existing
    token = secrets.token_urlsafe(32)
    os.environ["DEV_HELPERS_AUTOLOGIN_TOKEN"] = token
    return token
```

Plus: Django autoreloader działa na poziomie `runserver` polecenia, nie modułu — child re-importuje wszystko, więc AppConfig.ready() w child uruchamia się i znajdzie env var z parenta.

## 9. Dotfile management

### 9.1 Lifecycle

Dotfile'y są zapisywane w **AppConfig.ready()** — najwcześniejszy moment, gdy Django app jest gotowy i można wyznaczyć `BASE_DIR`. To jest **przed** startem HTTP serwera, więc plik jest na miejscu zanim runserver przyjmie pierwsze żądanie.

```python
import os

class DjangoDevHelpersConfig(AppConfig):
    name = "django_dev_helpers"

    def ready(self):
        from .conf import get_config
        cfg = get_config()
        if not cfg.is_active():
            return  # default off; brak side effects.

        # Hard fail FIRST — przed czymkolwiek innym (token, dotfile, browser).
        # Raisuje gdy DEBUG=False + serving HTTP. Patrz §4.3.
        from .safety import assert_safe_to_activate, emit_sanity_warnings, _is_serving
        assert_safe_to_activate(cfg)
        emit_sanity_warnings(cfg)  # ALLOWED_HOSTS, SECRET_KEY hints (warning-only)

        # Token init działa zawsze gdy autologin enabled — token w env musi
        # być spójny dla autoreloader-a i child requestów. Idempotentne.
        if cfg.autologin.enabled:
            from .tokens import init_token
            init_token()

        # Side effects niżej — tylko gdy faktycznie serwujemy HTTP. Inne
        # commands (migrate, test, shell) nie potrzebują dotfile'i ani
        # browser-open. Plus: w autoreloaderze odpalamy się TYLKO w child
        # process (RUN_MAIN=true) — parent jest tylko watcher-em.
        is_autoreload_parent = os.environ.get("RUN_MAIN") != "true"
        if not _is_serving():
            return
        if is_autoreload_parent and "runserver" in sys.argv:
            # Parent watcher — child wykona side effects po reload-zie.
            return

        if cfg.dotfiles.enabled:
            from .dotfiles import write_all_dotfiles, register_cleanup
            write_all_dotfiles(cfg)
            register_cleanup(cfg)

        if cfg.gitignore.mode != "off":
            from .gitignore import check_gitignore
            check_gitignore(cfg)  # warn / auto-add / error / off

        if cfg.agent_help.auto_print:
            from .prompt import register_first_request_print
            register_first_request_print(cfg)

        if cfg.browser_open.enabled:
            from .browser import spawn_self_probe_thread
            spawn_self_probe_thread(cfg)
```

**Kolejność jest istotna:**

1. `is_active()` early-exit dla wyłączonego pakietu.
2. `assert_safe_to_activate()` przed *czymkolwiek*: gdy raise — Django nie startuje, brak partial state.
3. Token init — przed serving guard, bo autologin view ma działać też w child autoreloadera.
4. Serving guard — odcina dotfile'y, gitignore, agent help, browser dla non-serving commands (migrate / test / shell etc.).
5. RUN_MAIN guard — `runserver` tworzy parent watcher process (z `RUN_MAIN` nieustawionym) i child worker process (z `RUN_MAIN=true`). Side effects tylko w child — inaczej browser-open, agent help, dotfile cleanup uruchamiałyby się dwa razy. Token init jest powyżej guarda, bo musi być w env parent-a, żeby child go zdziedziczył.

`gunicorn`/`uwsgi`/`uvicorn` nie mają `RUN_MAIN` semantyki — `is_autoreload_parent` jest `True` (bo env var nieustawiony), ale check `"runserver" in sys.argv` jest `False`, więc kod przechodzi dalej. Side effects działają normalnie.

### 9.2 Pliki

| Plik | Zawartość | Sekret | Source |
|---|---|---|---|
| `.dev_helpers_token` | autologin token (text) | tak | `DEV_HELPERS_AUTOLOGIN_TOKEN` env lub generated |
| `.dev_helpers_port` | port runservera (int) | nie | `DEV_HELPERS_PORT` env lub `request_started` signal lub `sys.argv` parse |
| `.dev_helpers_pg_host` | host-side PG hostname | nie | `DEV_HELPERS_DB_HOST` env lub `settings.DATABASES['default']['HOST']` |
| `.dev_helpers_pg_port` | host-side PG port | nie | `DEV_HELPERS_DB_PORT` env lub `settings.DATABASES['default']['PORT']` |
| `.dev_helpers_redis_host` | host-side Redis hostname | nie | `DEV_HELPERS_REDIS_HOST` env lub parse `settings.CACHES['default']['LOCATION']` |
| `.dev_helpers_redis_port` | host-side Redis port | nie | `DEV_HELPERS_REDIS_PORT` env lub parse `settings.CACHES['default']['LOCATION']` |

Host i port są w **oddzielnych** plikach (nie `host:port` w jednym), bo to ułatwia agentowi: `psql -h $(cat .dev_helpers_pg_host) -p $(cat .dev_helpers_pg_port)` bez parsowania. W praktyce host = `localhost` lub `127.0.0.1` w 99% scenariuszy, ale standalone tryby (Docker w VM, ngrok, LAN) wymagają jawnego host-a.

### 9.3 Discovery `runserver` portu i hosta

Problem: AppConfig.ready() jest wywołane przed `runserver` parsuje argv. Można:

1. **Parse `sys.argv`** — szukamy "runserver" + następnego argumentu typu `host:port` albo `port`. Działa dla `manage.py runserver 49152` i `manage.py runserver 127.0.0.1:49152`. Default Django port 8000 jeśli brak.
2. **Env var `DEV_HELPERS_PORT`** — jawnie ustawione przez orchestrator.
3. **Signal `request_started`** — pierwsza request odsłania `request.get_host()` z portem; jednorazowa aktualizacja dotfile'a.

Strategia: env var (jeśli ustawiony) > sys.argv parse > signal fallback (aktualizacja po pierwszym requeście).

**Host w prompt template:** gdy bind jest `0.0.0.0` (np. `runserver 0.0.0.0:8000`), prompt template **nie używa** `0.0.0.0` jako host (browser nie podłączy się do `0.0.0.0`). Substytucja:

- `0.0.0.0` → `localhost`
- `::` (IPv6 any) → `localhost`
- inny bind (np. `127.0.0.1`, konkretny IP, hostname) → użyj tak jak jest

Toggle override: `DJANGO_DEV_HELPERS["agent_help"]["display_host"]` (default `None` = auto-substytucja).

### 9.4 Cleanup

Atexit handler usuwa dotfile'y. Plus signal handler na `SIGTERM` (Django runserver dostaje SIGTERM przy Ctrl-C; default `signal.signal(SIGTERM, ...)`).

Stale dotfiles z poprzedniego, ubitego run-u: AppConfig na starcie nadpisuje (write, nie append) i kasuje przy następnym SIGTERM.

`django-run-site` (CLI orchestrator) **nie pisze** dotfile'i ani ich nie sprząta — to jest wyłącznie domena dev-helpers. Brak double-write/double-cleanup po stronie CLI.

**Chmod 600 na `.dev_helpers_token`:** best-effort. Na POSIX (Linux, macOS) działa zawsze. Na Windows `os.chmod` ma ograniczone semantyki (filesystem ACL nie jest mapowane 1:1) — write operuje, ale token nie jest naprawdę "secret" filesystem-wise. Spec nie supportuje Windows officially w v0.1, ale chmod failure → `logger.warning(...)`, nie error.

### 9.5 Project root discovery (dotfile directory)

`dotfiles.directory = None` (default) → discovery:

1. `os.environ["DEV_HELPERS_PROJECT_ROOT"]` (jeśli ustawione przez orchestrator).
2. **Walk-up** od `Path(settings.BASE_DIR)` w górę: pierwszy katalog zawierający `manage.py` lub `pyproject.toml` lub `runsite.toml` lub `.git/`.
3. Fallback: `Path(settings.BASE_DIR)` (sam BASE_DIR, **nie** parent).

```python
# project_root.py
def resolve_project_root(cfg) -> Path:
    if cfg.dotfiles.directory:
        return Path(cfg.dotfiles.directory).expanduser().resolve()
    env_value = os.environ.get("DEV_HELPERS_PROJECT_ROOT")
    if env_value:
        return Path(env_value).resolve()
    from django.conf import settings
    base_dir = Path(settings.BASE_DIR).resolve()
    markers = {"manage.py", "pyproject.toml", "runsite.toml", ".git"}
    for candidate in [base_dir, *base_dir.parents]:
        if any((candidate / m).exists() for m in markers):
            return candidate
    return base_dir
```

Powód: `Path(BASE_DIR).parent` jako default jest niebezpieczny — w flat layout (`BASE_DIR = ~/projects/myapp`) parent to `~/projects/`, dotfile'y trafiają do home dir-a, nie do repo. Walk-up szuka markerów projektu (manage.py, pyproject.toml, runsite.toml, .git) i zatrzymuje się na pierwszym dopasowaniu — zarówno dla `src/` layout (BASE_DIR jest niżej niż project root, walk-up znajdzie git root) jak i flat layout (BASE_DIR sam jest project root, marker `manage.py` w nim jest).

### 9.6 Discovery DB/Redis info

Default `lookup.source = "auto"`:

```python
def resolve_pg_endpoint(cfg) -> tuple[str, int] | None:
    if cfg.lookup.source in ("env", "auto"):
        host = os.environ.get("DEV_HELPERS_DB_HOST")
        port = os.environ.get("DEV_HELPERS_DB_PORT")
        if host and port:
            return host, int(port)
    if cfg.lookup.source in ("settings", "auto"):
        from django.conf import settings
        db = settings.DATABASES.get("default", {})
        host = db.get("HOST") or "127.0.0.1"
        port = db.get("PORT")
        if port:
            return host, int(port)
    return None
```

Redis: parse `settings.CACHES['default']['LOCATION']` jako URL (`redis://host:port/N`) lub env var `DEV_HELPERS_REDIS_HOST`/`DEV_HELPERS_REDIS_PORT`. Jeśli ani jednego — skip dotfile'a, nie blokuje reszty.

Custom: `lookup.callable = "myproject.dev_helpers_lookup:resolve"` zwraca dict `{"pg_host": ..., "pg_port": ..., "redis_host": ..., "redis_port": ...}`. Override pełny.

## 10. Token generation (autologin)

`tokens.py`:

```python
import os
import secrets

ENV_VAR = "DEV_HELPERS_AUTOLOGIN_TOKEN"
TOKEN_BYTES = 32

def init_token() -> str:
    existing = os.environ.get(ENV_VAR)
    if existing:
        return existing
    token = secrets.token_urlsafe(TOKEN_BYTES)
    os.environ[ENV_VAR] = token
    return token

def current_token() -> str | None:
    return os.environ.get(ENV_VAR)
```

`init_token()` jest idempotentne — ten sam token w parent i child autoreloader.

## 11. Agent help / prompt template

### 11.1 Template renderer

Default template (English; user może override):

```text
─── django-dev-helpers — agent prompt ───────────────────────────────

Server is up at: http://{host}:{port}

To fetch the authenticated home page (autologin):
  T=$(cat "{token_path}")
  PORT=$(cat "{port_path}")
  J=$(mktemp)
  curl -sc "$J" -L "http://{host}:$PORT/{autologin_path}?token=$T" >/dev/null
  curl -sb "$J" "http://{host}:$PORT/<path>"
  rm "$J"

To connect to PostgreSQL ({db_user_or_redacted}):
  PG_HOST=$(cat "{pg_host_path}")
  PG_PORT=$(cat "{pg_port_path}")
  {pg_command_with_or_without_password}

To connect to Redis:
  REDIS_HOST=$(cat "{redis_host_path}")
  REDIS_PORT=$(cat "{redis_port_path}")
  redis-cli -h "$REDIS_HOST" -p "$REDIS_PORT"

──────────────────────────────────────────────────────────────────────
```

`pg_command_with_or_without_password` zależy od `agent_help.show_db_credentials`:

- `True` (default): `PGPASSWORD={db_password} psql -h "$PG_HOST" -p "$PG_PORT" -U {db_user} -d {db_name}`
- `False`: `psql -h "$PG_HOST" -p "$PG_PORT" -U {db_user} -d {db_name}  # password from env / .pgpass / settings`

Hosty są czytane z dotfile'i (nie hardcode `localhost`), żeby standalone scenarios (LAN, ngrok, Docker w VM) działały.

User może override przez `DJANGO_DEV_HELPERS["agent_help"]["template"] = "<custom>"` (string z placeholders).

### 11.2 Auto-print przy starcie

`agent_help.auto_print = True` (default) → AppConfig rejestruje `request_started` one-shot handler, który po pierwszym requeście drukuje template do stdout (ten sam stream co runserver logs). To gwarantuje, że banner pojawia się dopiero gdy serwer realnie odpowiada — żaden race z autoreloaderem.

### 11.3 Management command

`python manage.py dev_helpers_print_help` — drukuje template natychmiast (bez czekania na request). Użycie: w skryptach CI sanity-check, lub gdy user chce skopiować banner ponownie.

### 11.4 `dev_helpers_doctor` — pełna diagnoza

`python manage.py dev_helpers_doctor` — domyślnie sprawdza **wszystko** (full check):

- konfiguracja `DJANGO_DEV_HELPERS` walidacja typów,
- aktywacja: `enabled` flag, `DEBUG=True` check (raisuje gdy `DEBUG=False`),
- autologin config: `user_lookup_field`/`user_lookup_value` rozpoznane przez user model, `auth_backend` import-able,
- existence usera w bazie (lookup query),
- `settings.DATABASES` reachability — `psql` connect ping (lub `psycopg.connect` short timeout),
- `settings.CACHES` Redis reachability — `redis-cli PING` lub Redis client connect,
- token env var: jest? rozsądna długość?,
- dotfile dir writable (próba `Path.touch()` + `unlink()`),
- `.gitignore` (mode-zależnie),
- `ALLOWED_HOSTS` sanity,
- `SECRET_KEY` sanity.

Flagi:

```bash
python manage.py dev_helpers_doctor              # full check (default)
python manage.py dev_helpers_doctor --skip-services  # bez DB/Redis ping
python manage.py dev_helpers_doctor --json       # output JSON dla CI
```

Exit code: 0 = wszystko OK; 1 = errors; 2 = warnings only. CI-friendly.

## 13. Współpraca z `django-run-site`

### 13.1 Kontrakt env-var

`django-run-site` (CLI orchestrator) ustawia env vars dla procesu runservera:

| Env var | Set by `django-run-site` | Read by `django-dev-helpers` |
|---|---|---|
| `DEV_HELPERS_AUTOLOGIN_TOKEN` | wygenerowany token | `tokens.init_token()` reuse |
| `DEV_HELPERS_PORT` | port runservera (CLI go wybiera) | `dotfiles.discover_port()` |
| `DEV_HELPERS_DB_HOST` | host-side host PG (zwykle `127.0.0.1` lub `localhost`) | `dotfiles.resolve_pg_endpoint()` |
| `DEV_HELPERS_DB_PORT` | host-side port PG | jw. |
| `DEV_HELPERS_REDIS_HOST` | host-side host Redis | `resolve_redis_endpoint()` |
| `DEV_HELPERS_REDIS_PORT` | host-side port Redis | jw. |
| `DEV_HELPERS_PROJECT_ROOT` | absolute path do project root | `dotfiles.resolve_directory()` |

`django-dev-helpers` w trybie standalone:

- Jeśli env var nie ustawiony — fallback do `settings` lub auto-detection.
- Wszystkie env-y są opcjonalne; pakiet nie wymaga `django-run-site`.

### 13.2 Browser open ownership

`dev-helpers` default: `browser_open.enabled=True` → otwiera **autologin URL** (z tokenem).
`run-site` default: open homepage (kontroluje `--no-browser`).

| Tryb | Co się otwiera |
|---|---|
| Z `django-run-site`, defaultowo | 2 zakładki: homepage (CLI) + autologin URL (dev-helpers) |
| Z `django-run-site`, `--no-browser` | 1 zakładka: autologin URL (dev-helpers) |
| Z `django-run-site`, `--no-browser` + `dev-helpers browser_open.enabled=False` | 0 zakładek |
| Standalone (`browser_open.enabled=True`, default) | 1 zakładka: autologin URL (dev-helpers self-probe thread) |
| Standalone (`browser_open.enabled=False`) | 0 zakładek |

Świadomy wybór: każdy pakiet odpowiada za jedno. CLI nie zna autologin URL path. Dev-helpers nie zna kontekstu orchestratora. Razem dają user-friendly default; każdy z osobna jest deterministic.

### 13.3 Banner ownership

| Co | Owner |
|---|---|
| Banner z portami PG/Redis, dump info, celery status | `django-run-site` (CLI stdout) |
| Agent help / prompt template | `django-dev-helpers` (Django stdout, mux'owany przez CLI) |

Te dwa banner-y są drukowane oddzielnie i w różnych momentach: orchestrator banner przy starcie, agent help dopiero po first successful request. Brak duplikacji.

## 15. Migracja BPP do `django-dev-helpers`

Krok niezależny od `django-run-site` migracji. Można zrobić **przed** adopcją CLI orchestratora, dając wartość natychmiast.

1. Zainstaluj: `uv add django-dev-helpers --group dev`.
2. Dodaj do `INSTALLED_APPS` w `django_bpp/settings/base.py`.
3. Dodaj `DJANGO_DEV_HELPERS` config z BPP-specific cookie:

```python
# django_bpp/settings/base.py
DJANGO_DEV_HELPERS = {
    "enabled": os.environ.get("DJANGO_DEV_HELPERS_ENABLED") == "1",
    "autologin": {
        # cookielaw_accepted: BPP używa django-cookielaw; bez tego
        # cookie banner zasłania UI i agent musi je dismiss-ować
        # przy każdym requeście. W dev to czysto irytujące.
        "extra_cookies": [
            {
                "name": "cookielaw_accepted",
                "value": "1",
                "max_age": 31536000,  # 1 rok
                "samesite": "Lax",
                "path": "/",
                "secure": False,
                "httponly": False,
            },
        ],
    },
}
```
4. W `django_bpp/urls.py`:
   ```python
   from django_dev_helpers.urls import autologin_urlpatterns
   urlpatterns = [*autologin_urlpatterns(), ...]
   ```
5. Usuń `django_bpp/views_run_site_autologin.py` po e2e teście.
6. Zaktualizuj `.gitignore` na nowe nazwy dotfile'i (jeśli były `.run_site_*`, są teraz `.dev_helpers_*`).
7. Update CLAUDE.md / agent instructions na nowe ścieżki dotfile'i.

`bpp/management/commands/run_site.py` w tym kroku jeszcze działa — używa starych env-var-y i sam zapisuje swoje dotfile'y. Nazewnictwo na chwilę jest mieszane (`.run_site_*` od starego command-a + `.dev_helpers_*` od dev-helpers AppConfig). To akceptowalne na czas migracji; finalny stan po przejściu na `django-run-site` v0.3 — tylko `.dev_helpers_*`.

Zero hard deps poza Django. To jest pakiet "pure Django app" z minimum surface.

## 18. Rozstrzygnięte decyzje

Wszystkie decyzje projektowe są twarde — pakiet nie ma toggle-i pomiędzy nimi.

1. **`DEBUG=False` + `enabled=True` → `ImproperlyConfigured`.** Brak override. §4.3.

2. **Custom user model lookup w v0.1.** Pola `user_lookup_field` + `user_lookup_value` w `[autologin]` configu. Default: `username="admin"`. §7 + §8.2.

3. **`dev_helpers_doctor` sprawdza wszystko domyślnie** — w tym DB/Redis reachability. Opt-out przez `--skip-services`. JSON output dla CI przez `--json`. §11.4.

4. **Token: `secrets.token_urlsafe(32)`.** Krótszy niż `token_hex`, bezpieczny w URL query string. §10.

5. **Token tylko w query string** — nigdy nie expose w cookie (custom header, response body etc.). Po zalogowaniu — standardowa Django session. §8.2.

## 19. Podsumowanie

`django-dev-helpers` to mała, samodzielna Django app, która rozwiązuje trzy konkretne dev-time problemy:

- agenty LLM mogą logować się automatycznie via dotfile token,
- developer ma czytelny prompt z curl/psql/redis-cli wzorcami,
- `.gitignore` jest sanity-checkowany.

Pakiet jest **default-off na produkcji** (kill switch przez explicit `enabled=True`), ma defense-in-depth w widoku, i nie wymaga `django-run-site`. Współpraca z orchestratorem przez kontrakt env-var jest przezroczysta — pakiet preferuje env, ale fallback do `settings` zawsze działa.
